# P0.15 — Long Document Support (Active)

**Status:** active, supersedes the deferred `05-deferred-document-chunking.md`
**Effort:** ~3-4 days
**Lands before:** P9 (MCP API) — so the search endpoint can ship multi-tier from day 1
**Trigger:** scope cap raised past 40 pages; first workspace expected to ingest manuals + long contracts

## What changed

Earlier we capped scope at 40 pages and deferred chunking. New decision:

- Cover documents up to ~300 pages cleanly
- Support tree-walk reasoning (PageIndex-style) for multi-section
  questions
- Keep single-vector path for short content (chat/email/meeting/note)

## Architectural approach — three tiers + tree

```
cortex_entities         <-- 1 row per doc (root); doc-level embedding (P0.14)
cortex_sections (NEW)   <-- N rows per long doc; chapter/section embedding + LLM summary
cortex_chunks (NEW)     <-- M rows per section; chunk embeddings (needle-in-haystack)
```

Three tiers handle three different queries:

| Tier | Query | Retrieval |
|---|---|---|
| **1. Doc-level** | "which document is about X?" | ANN over `cortex_entities.doc_embedding` |
| **2. Section-level** | "which chapter of the manual?" | ANN over `cortex_sections.section_embedding` |
| **3. Chunk-level** | "exact passage that says Y" | ANN over `cortex_chunks.chunk_embedding` |
| **+ Tree walk** | "what does Ch7 imply for Ch12?" | agent reads section summaries, picks path, recurses |

Section summaries also enable PageIndex-style tree-walk reasoning
at read time without forcing it as the only mode.

## When tiers activate

| Doc length | Tier 1 (doc) | Tier 2 (sections) | Tier 3 (chunks) |
|---|---|---|---|
| ≤ 4k tokens (~5 pages) | ✅ | ❌ | ❌ |
| 4k–32k tokens (~5–40 pages) | ✅ | ❌ | ✅ |
| > 32k tokens (~40+ pages) | ✅ | ✅ | ✅ |

Thresholds configurable per TypeSpec.

## Schema

### `cortex_sections`

```python
class CortexSection(TimestampsMixin):
    id                 = UUIDField PK
    entity             = FK CortexEntity on_delete=CASCADE related_name="sections"
    parent_section     = UUIDField null=True   # tree structure (self-ref, no FK)
    depth              = IntegerField           # 0=root, 1=H1, 2=H2, ...
    section_idx        = IntegerField           # ordering within parent
    section_path       = CharField(500)         # "Chapter 7 > Warranty > Exclusions"
    title              = CharField(500)         # heading text
    byte_start         = IntegerField
    byte_end           = IntegerField
    token_count        = IntegerField
    summary            = TextField              # HaikuFitter-generated, ~200 chars
    section_embedding  = VectorField(384)
    # No body — load via entity.body[byte_start:byte_end]
```

Indexes:
- `(entity, section_idx)` unique
- `(entity, depth)` btree
- IVFFLAT on `section_embedding`

### `cortex_chunks`

```python
class CortexChunk(TimestampsMixin):
    id                 = UUIDField PK
    entity             = FK CortexEntity on_delete=CASCADE related_name="chunks"
    section            = FK CortexSection on_delete=CASCADE null=True related_name="chunks"
    chunk_idx          = IntegerField           # global within entity
    byte_start         = IntegerField
    byte_end           = IntegerField
    token_count        = IntegerField
    section_path       = CharField(500)         # denormalised for fast filters
    chunk_embedding    = VectorField(384)
    text_preview       = CharField(500)         # first ~500 chars for debug
```

Indexes:
- `(entity, chunk_idx)` unique
- IVFFLAT on `chunk_embedding`

## Read API — `cortex.search`

```
POST /cortex/v1/search
{
  "workspace_id": "...",
  "query": "warranty period",
  "scope": {client_id, project_id},
  "top_k": 10,
  "mode": "auto",          # auto | chunk | section | tree-walk
  "include_neighbours": 1, # also load N±1 chunks
}
```

Modes:

| Mode | Strategy |
|---|---|
| `auto` | doc-level ANN → if hit has chunks, chunk-level ANN within entity → return chunks |
| `chunk` | global chunk-level ANN scoped to workspace |
| `section` | section-level ANN → return sections (for "which chapter?") |
| `tree-walk` | LLM-driven traversal of section summaries (PageIndex-style) |

Response uniform:

```json
{
  "hits": [
    {
      "entity_id": "...",
      "entity_title": "Foo Widget Manual",
      "entity_url": "https://...",
      "score": 0.87,
      "matched_chunks": [
        {
          "chunk_idx": 144,
          "section_path": "Chapter 7 > Warranty Terms",

          "byte_end": 78891,
```

## PageIndex-style tree walk (mode = `tree-walk`)

When the agent needs multi-section reasoning ("what does Chapter 7
imply for Chapter 12 conclusions?"), single-vector ANN can miss the
chain. Tree walk handles it:

```python
def tree_walk(entity_id, question, max_depth=4, branch_factor=3):
    root_sections = CortexSection.objects.filter(
        entity_id=entity_id, depth=1,
    ).order_by("section_idx")

    chosen_path = []
    candidates = list(root_sections)
    for depth in range(max_depth):
        if not candidates:
            break
        # LLM picks top-N children based on title + summary
        prompt = build_prompt(question, candidates)
        choices = llm.choose(prompt, k=branch_factor)
        chosen_path.append(choices)
        # Descend to children of chosen sections
        candidates = CortexSection.objects.filter(
            parent_section__in=[c.id for c in choices],
        )
    return chosen_path  # leaf sections → load their bodies for synthesis
```

Cost: ~max_depth × branch_factor LLM calls per query
(e.g. 4×3 = 12 ≈ $0.005 with Haiku).

When to use:
- Multi-hop reasoning queries
- Long manuals where chunk ANN returns unrelated passages
- `mode=tree-walk` opt-in per query

## Section summaries — write-time cost

Per 100-page manual:
- ~30 sections (H1/H2 boundaries)
- 30 × ~$0.0003 Haiku summary calls = ~$0.01 per ingest
- Time: ~5-10s for summarisation (Haiku latency)
- Total ingest: ~15-20s per 100-page manual

Acceptable; one-time cost.

## Cost analysis at scale

Workspace with 1000 long-form docs (avg 50 pages each):
- Storage: ~5GB markdown in S3 + ~5GB PG (chunks + sections + vectors)
- Embeddings: ~50,000 chunks + ~30,000 sections + ~1000 docs = ~81,000 BGE-small calls (free, CPU)
- Summaries: ~30,000 Haiku calls = ~$10 total one-time
- Per query at read: 1 question embed + 2-3 ANN searches + 1 LLM synthesis = ~$0.005

## What stays single-vector

These types keep current behaviour (doc-level only, no sections/chunks):

- `chat`, `email`, `meeting`, `note`, `clip`, `ticket`
- `person`, `org`, `project`, `concept`, `decision`

Only `doc` activates the long-doc path, and only above thresholds.
